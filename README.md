# Мультиагентная система для Data Science

Автономная мультиагентная система, которая решает задачи машинного обучения на табличных данных. Работает как Claude Code: генерирует код, видит результат, рассуждает, генерирует ещё код — и так до достижения цели.

## Структура репозитория

```
├── .env                    # Конфигурация LLM (модель, API ключ, URL)
├── requirements.txt        # Зависимости (openai, python-dotenv, pinned)
├── main.py                 # Точка входа
├── agents/
│   ├── __init__.py
│   ├── base.py             # Базовый агент: LLM клиент, парсинг, мониторинг
│   ├── orchestrator.py     # Оркестратор: мультитурновый ReAct цикл
│   ├── data_agent.py       # Агент предобработки данных
│   ├── model_agent.py      # Агент обучения модели
│   ├── eval_agent.py       # Агент оценки и обратной связи
│   ├── guardrails.py       # Проверка безопасности кода перед исполнением
│   └── validators.py       # Валидация входов/выходов агентов
├── benchmarking/
│   ├── tracker.py          # Трекер экспериментов (сохранение метрик)
│   └── compare.py          # Сравнение результатов между запусками
├── report/
│   ├── models.md           # Обоснование выбора LLM
│   └── architecture.md     # Архитектура системы с диаграммами
├── rag.py                  # BM25 RAG по документации библиотек
├── prompt.txt              # Описание задачи
└── data/                   # Данные в формате Kaggle
```

## Архитектура

Подробное описание с диаграммами: [`report/architecture.md`](report/architecture.md)

### Общий принцип

Система работает в два этапа:

**1. Планирование (Plan Phase)**
- Читает задачу из `prompt.txt`
- Профилирует данные (типы, пропуски, распределение)
- Проверяет ресурсы системы (CPU, RAM, платформа)
- Устанавливает ML библиотеки и индексирует их документацию в RAG
- Вся информация передаётся в LLM для формирования плана

**2. Исполнение (ReAct Loop)**
```
while время не вышло:
    ответ LLM → содержит блок кода?
    ├── да  → guardrails проверка → выполнить код → показать результат → продолжить
    └── нет → агент закончил работу
```

Оркестратор ведёт мультитурновый диалог с LLM. Вся история (код, результаты, ошибки) накапливается в контексте. Агент свободно решает что делать дальше — нет фиксированного пайплайна.

### Агенты (4 шт.)

**OrchestratorAgent** — мозг системы:
- Мультитурновый диалог с LLM
- Свободно решает стратегию: что предобработать, какую модель обучить, когда остановиться
- Может вызывать любого специалиста в любом порядке, любое количество раз
- Захватывает stdout исполненного кода и показывает агенту

**DataAgent** — предобработка данных:
- LLM генерирует код предобработки (pandas, numpy)
- Сам устанавливает нужные библиотеки
- Возвращает числовые DataFrame

**ModelAgent** — обучение модели:
- LLM генерирует код обучения (lightgbm, xgboost, optuna)
- Принимает time_budget для контроля времени
- Автоматическое исправление кода при ошибках (до 5 попыток)

**EvalAgent** — оценка качества:
- Вычисляет метрики (MSE, RMSE, MAE, R²) без внешних зависимостей
- LLM анализирует результаты и даёт рекомендации
- Формирует feedback для следующей итерации

### RAG модуль (`rag.py`)

Индексирует **реальную документацию** установленных библиотек:
- Извлекает docstring из всех публичных классов, функций и методов через `inspect`
- Каждый документ содержит имя пакета, версию, сигнатуру и описание
- BM25 поиск (k1=1.5, b=0.75)
- Индексируются: pandas, numpy, scikit-learn, lightgbm, xgboost, optuna

### Мониторинг и бенчмаркинг

- Каждый вызов LLM логируется: временная метка, задержка, токены, успех/ошибка
- `ExperimentTracker` сохраняет метрики каждой итерации в `benchmarking/runs/<run_id>.json`
- `compare.py` позволяет сравнить результаты между запусками
- Финальный отчёт: количество вызовов, средняя задержка, токены по каждому агенту

### Безопасность

- **Guardrails** (`agents/guardrails.py`): проверка кода на опасные паттерны перед `exec()`
- **Restricted builtins**: удалены `eval`, `exec`, `compile` из namespace
- **Input validation** (`agents/validators.py`): валидация CSV, предсказаний, выходов агентов
- Парсинг ответов с обработкой `<think>` тегов Qwen3
- Автоматическое исправление кода при ошибках исполнения (до 5 попыток)
- Контроль времени (time budget)

## Предварительные требования

- Python 3.10+
- API ключ для OpenAI-совместимого LLM сервиса

## Запуск

```bash
# 1. Создать виртуальное окружение
python -m venv .venv
source .venv/bin/activate

# 2. Установить зависимости
pip install -r requirements.txt

# 3. Настроить .env
cp .env.example .env  # или создать вручную (см. ниже)

# 4. Запустить
python main.py
```

Агент автоматически:
1. Прочитает задачу из `prompt.txt`
2. Проверит ресурсы системы
3. Установит и проиндексирует ML библиотеки
4. Спланирует подход
5. Будет итеративно улучшать модель, пока не достигнет цели или не истечёт время
6. Сохранит лучший результат в `submission.csv`
7. Сохранит бенчмарк в `benchmarking/runs/`

### Конфигурация

По умолчанию используется модель **Qwen 3 Coder Next** (`qwen/qwen3-coder-next`) через API vsellm.ru.
Обоснование выбора модели: [`report/models.md`](report/models.md)

`.env` файл:
```
MODEL=qwen/qwen3-coder-next
API_KEY=your-api-key
BASE_URL=https://api.vsellm.ru/v1
```

### Сравнение запусков

После нескольких запусков можно сравнить результаты:
```bash
python -m benchmarking.compare
```

### Смена задачи

Система не привязана к конкретному датасету. Для другой задачи:
1. Положить данные в `data/`
2. Отредактировать `prompt.txt`
